## Confirmed findings (kept tests)

### F-VAULT-FRAG (MEDIUM) — **FIXED** — vault account not pinned to canonical address -> honest-user LoF via liquidity fragmentation
- **Component:** WRAPPER `src/v16_program.rs`. **Test (GREEN):** `v16_exploit_vault_fragmentation_strands_honest_withdraw`.
- **RCA:** vault_token validated only by `owner == vault_authority PDA` (verify_vault_token_account ~:11537 used by handle_deposit:5695; verify_withdrawable_token_accounts:11510 used by handle_withdraw:5749). No canonical vault address is pinned (config stores `vault: u128` balance, not a pubkey; InitMarket records none). ANY vault-authority-owned token account is accepted.
- **Exploit (confirmed vs BPF):** honest deposits 1M to canonical; attacker deposits 500k routed to a self-created vault-authority-owned `fake_vault` (accepted), then withdraws 500k from the CANONICAL vault (honest funds). Canonical now 500k < c_tot 1M -> honest user`'`s full 1M withdraw FAILS; 500k stranded in fake_vault.
- **Severity nuance:** USER-reachable, custody account, confirmed honest LoF; BUT self-sacrificial 1:1 (attacker abandons equal funds to strand) and reversible (recovering un-strands) -> griefing DoS / accidental-fragmentation hazard, not profitable theft.
- **Affected handlers:** deposit, withdraw, withdraw_insurance, withdraw_backing_bucket(+earnings), close_resolved, claim_resolved_payout_topup, cure (all validate vault by owner only).
- **Fix (LANDED):** pinned the vault to the canonical ATA of (vault_authority, mint) inside the shared verify_vault_token_account + verify_withdrawable_token_accounts helpers (covers all 14 vault-touching call sites: deposit/withdraw/insurance/backing(+earnings)/swap/close_resolved/claim). `canonical_vault_address` derives the ATA; any other vault_authority-owned account is now rejected. Harness updated to use the ATA vault; regression `v16_regression_vault_pinned_to_canonical_ata_no_fragmentation` confirms deposit/withdraw to a non-canonical vault rejects and honest withdrawals are never stranded. Full suite green (119). Writeup: /tmp/FINDING-vault-fragmentation.md.

### F-TRADENOCPI-FEE (MEDIUM) — **FIXED (2026-06-01)** — TradeNoCpi fee was evadable via an unvalidated caller-supplied `exec_price`
- **Layer:** PROGRAM / wrapper (`src/v16_program.rs`) — a **missing validation**, not an engine math bug.
- **RCA:** `handle_trade_nocpi_zero_copy` (src/v16_program.rs:5796) takes the caller-supplied `exec_price` and passes it straight into `TradeRequestV16 { exec_price, .. }` (:5846/:5849) **without validating it against the asset mark**. The engine then computes the trade fee on that value: `notional = trade_notional_floor(size_q, exec_price) = size_q*exec_price/POS_SCALE` (percolator/src/v16.rs:10429, 12473), `fee = notional*fee_bps` (:10430). The **position entry and initial margin use `effective_price` (the mark), NOT exec_price** — so exec_price's ONLY effect is the fee basis. **TradeCpi does NOT have this gap**: the wrapper pins the matcher's price to the oracle (`ret.exec_price_e6 != oracle_price_e6` -> reject, src/v16_program.rs:3640), so the same engine fee formula is correct there. The defect is the asymmetry: TradeNoCpi omits the equivalent check.
- **Exploit (confirmed vs BPF):** with mark=100, a POS_SCALE×1000 position at `exec_price=1` is charged fee **20**, vs **2000** at `exec_price=100` (the mark) — a **100× underpayment** for an identically-sized, mark-valued position. Any two cooperating accounts (or one entity with two portfolios) trade at full mark value while paying ~1% of the intended fee; a single entity can wash-trade near-fee-free.
- **Severity:** MEDIUM — NOT a fund-drain (entry/margin are mark-based, no off-market PnL, no deposit theft; see `v16_attack_off_market_exec_price_wash_trade_prints_nothing` which confirms zero value is printed). It is a **fee/revenue-integrity** bug: trading fees fund the insurance backstop, so systematic evasion starves the solvency buffer.
- **Proposed fix (wrapper, percolator-prog):** in `handle_trade_nocpi_zero_copy`, before building the `TradeRequestV16`, force the fee basis to the mark — either (A) reject when `exec_price != effective_price` (mirrors TradeCpi's oracle pin; tightens TradeNoCpi to mark-execution; breaks the few tests that pass exec_price≠mark), or (B) overwrite `exec_price = effective_price` for the engine call so the fee is always computed on the mark notional (no test breakage; exec_price becomes a no-op it already effectively is). Recommend (B) unless exec_price is meant to be a slippage-limit (in which case keep it as a limit check but still compute the fee on the mark).
- **Fix (LANDED, approach B):** in `handle_trade_nocpi_zero_copy` (src/v16_program.rs:~5837) the fee basis is now pinned to the asset mark — `fee_basis_price = group.markets[asset_index].engine.asset.effective_price` is passed to `hybrid_trade_fee_bps_view` and into `TradeRequestV16.exec_price`, so the engine bills the fee on the price the trade actually settles at. The ORIGINAL caller `exec_price` is still forwarded to `update_hybrid_mark_after_trade_view` (a distinct role: in hybrid mode it is the reported trade price that drives mark discovery, EWMA/clamp-bounded), so hybrid price discovery is unchanged.
- **Design note discovered during fix:** the old behavior was *enshrined* by `v16_bpf_tradenocpi_executes_and_is_bounded`, which traded at exec_price=150 (mark 100) and asserted insurance=30 (fee on the consented price) while also asserting the position still entered at the mark. Since exec_price has NO settlement role (entry/margin are mark-based — verified by `v16_attack_off_market_exec_price_wash_trade_prints_nothing`), billing the fee on it was the defect; that test now asserts insurance=20 (fee on the mark). Per the user directive "we can't have 2.5 bip fees" this is treated as a bug, not intended design.
- **Regressions (GREEN):** `v16_attack_tradenocpi_fee_cannot_be_evaded_via_exec_price` proves exec_price ∈ {1, 50, 100, 100000} all charge the identical mark-based fee; updated `v16_bpf_tradenocpi_executes_and_is_bounded`. BPF rebuilt (`cargo build-sbf --no-default-features`); full suite green (193). TradeCpi unaffected (already oracle-pinned at :3640).
